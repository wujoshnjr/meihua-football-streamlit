# 兩階段語義盲解卦

## 為什麼需要隔離

如果模型在解卦前已看見首選比分、Poisson 機率、規則排名與量化門檻，它很容易先接受既定答案，再從卦義中挑選能支持答案的句子。這是「事後合理化」，不是語義推演。v4.2 用兩次彼此隔離的模型請求，讓解卦與比分決策具有可驗證的先後順序。

## 第一階段：盲解卦

第一階段可見：

- 賽前文字、比賽名稱、體用設定與 90 分鐘範圍。
- 固定的體卦、用卦、本卦、互卦、動爻、轉象、變卦與前後五行關係。
- 八卦與六十四卦的自然象、足球語義、攻守方式、節奏、風險與標籤。
- 不含量化結論的來源證據與動爻時窗說明。

第一階段不可見：

- 候選比分、首選／第二選／第三選與完整比分網格。
- Poisson 機率、勝平負機率、足球 λ、實力分與量化維度分數。
- 零球／高比分等已計算閘門、總球錨點、規則排序與規則效果。
- 相似案例的歷史預測、實際比分與賽後校準。

第一階段必須輸出主局矛盾、本→互→動→變時間線、體用角色、兩方破門方式、收束機制、主劇本、替代劇本、反解、不確定性及被禁止的捷徑。它禁止輸出任何精確比分或機率。

## 第二階段：足球校準與比分決策

第二階段才接收：

- 第一階段已鎖定的盲解 JSON。
- 客觀足球實力、可信度、場地、純足球 λ 與 ±25% 有界卦象修正。
- 本地劇本的定性閘門與受限卦數訊號。
- 體勝、平局、用勝均有代表的中性排序候選池。
- 已確認且排除同場的歷史案例。

候選池不提供原規則順位，只提供比分、方向、總球、足球可行性帶與劇本原型。模型必須先列出採用的盲解劇本，再用足球資料提出支持或反證，最後選三個比分。每個比分理由必須能追溯到劇本、破門通道與足球校準。

## 失敗回退

- 第一階段 AI 失敗、第二階段仍可用時：使用本地語義骨架替代盲解，並在風險提示中明示。
- 第二階段失敗：保留已取得的盲解，但比分退回本地語義劇本與固定候選排序。
- AI 完全停用：仍顯示本地語義卦線、主解、反解與劇本分支；介面明確標示它是可重現模板，不宣稱是大型語言模型思考。

## 可稽核契約

測試會直接解析兩個 Prompt payload，確認第一階段沒有比分池、比分網格、足球先驗、攻守評分、量化門檻、規則排序或常見比分字串；並以 mock 模型驗證呼叫順序一定是「盲解 Schema → 決策 Schema」。CSV 保存盲解版本、完整盲解 JSON、核心論點、反解及第二階段採用的劇本。

## 限制

隔離可以降低答案錨定與事後合理化，但不能保證模型一定推理正確。GitHub Models 中不同模型的語義能力、上下文長度與配額不同；這套流程不宣稱能與任何特定對話模型完全等價。是否真正改善預測，仍須按版本用未參與規則制定的賽前鎖定案例做留出測試。

可用 `AI_DELIBERATION_MODEL` 與 `AI_MODEL` 分別設定第一階段盲解模型及第二階段決策模型。若前者未設定，兩階段使用同一模型；使用較小模型可能更快、更省額度，但仍較容易產生模板化敘事。
