# 系統架構

## 目標

v4 的首要目標是可驗證性：每一場預測都能重現「輸入 → 足球 λ → 卦象修正 → 候選 → AI → 鎖定 → 賽後評估」，並且能把純足球基線與卦象、AI 的增益分開。

## 信任邊界

| 區域 | 可做 | 不可做 |
|---|---|---|
| Python 起卦 | 計數、計算本互動變與體用 | 讀取本場實際比分 |
| 足球先驗 | 使用人工輸入的賽前實力、可信度、場地 | 使用卦象或支持方向加分 |
| 卦象引擎 | 在足球 λ 上作 ±25% 修正 | 取代足球基線或任意指定比分 |
| 歷史案例 | 使用已確認的舊賽果與校準 | 使用同一場比賽、未確認案例 |
| GitHub Models | 在固定候選池內做證據融合 | 改卦、創造候選池外比分、讀取本場賽果 |
| 賽後中心 | 回填結果與校準 | 改寫賽前輸入、卦象、λ、候選與雜湊 |

## 模組

- `version.py`：版本唯一來源。
- `models.py`：跨模組資料契約。
- `football_prior.py`：不含卦象的 λ 與 Poisson 網格。
- `meihua_engine.py`：確定性的字數與起卦。
- `score_engine.py`：連續卦線、規則治理、有界修正與比分排序。
- `evaluation.py`：AI 控制、結果解析與評估。
- `case_memory.py`：確認案例過濾、同場排除與相似度。
- `storage_v33.py`：v4 schema、鎖定、版本、遷移與 GitHub Contents。
- `ai_reasoner_v33.py`：結構化 Prompt、JSON schema 與失敗回退。
- `app_v33.py`：現行 UI；檔名保留作部署相容。

## 主要不變量

1. `MatchInput.scope` 固定為 90 分鐘，不含延長與 PK。
2. `HexagramResult` 只由三段起卦文字決定。
3. `football_prior` 不讀取任何卦象欄位。
4. `0.75 <= hexagram_multiplier <= 1.25`。
5. AI 分數必須存在於 `candidate_scores()` 的平衡候選池。
6. 同一預測內容產生同一 SHA-256 雜湊。
7. 雜湊不同的同場預測只能新增版本，不能覆寫。
8. 賽後更新只能修改結果、評估與校準欄位。

## 相容策略

舊版 CSV 可能只含 `首選比分／第二選比分／第三選比分`。載入時會映射到規則與最終欄位，但保留原欄位。舊資料標為 `legacy`，可以展示與檢索，不能計入 v4 鎖定盲測。
